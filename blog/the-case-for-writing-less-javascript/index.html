The case for writing less JavaScript

A tour of the patterns I've been reaching for lately — view transitions, declarative shadow DOM, and why I stopped using three framework features entirely.

I've been writing JavaScript professionally for over a decade. I'm good at it. And I've spent the last year trying to write as little of it as possible.

Not because JavaScript is bad. Because the web platform has gotten genuinely, substantively better, and I kept reaching for JavaScript to solve problems the browser now solves itself.

View Transitions

The View Transitions API landed in Chrome, then Safari, then Firefox. It lets you animate between page states — or actual page navigations — with a simple API wrapper and a few lines of CSS.

Before this, you needed a framework, a router, and a state management strategy just to do a fade between pages. Now:

document.startViewTransition(() => {
  document.querySelector('.content').innerHTML = newContent;
});

That's it. The browser handles the snapshot, the animation, the cleanup. CSS controls the timing and easing. The JavaScript footprint is three lines.

Declarative Shadow DOM

Declarative Shadow DOM lets you ship web components that are server-rendered without JavaScript. The shadow DOM attaches via HTML alone:

<my-button>
  <template shadowrootmode="open">
    <style>button { ... }</style>
    <button><slot></slot></button>
  </template>
  Click me
</my-button>

No customElements.define. No JS required for the initial render. Progressive enhancement falls out naturally.

The three framework features I stopped using

1. Client-side routing for simple sites. If you don't have truly dynamic state that needs to persist across navigations, you don't need a client-side router. MPA with view transitions is faster, simpler, and more resilient.

2. Component-level data fetching on page load. For most content, fetch it on the server. Return HTML. The component doesn't need to know about it.

3. Global state management for UI state. CSS custom properties, URL state, and localStorage cover 90% of what people reach for Redux or Zustand to handle. The 10% that remains is where a proper state library actually earns its place.

None of this is a polemic against frameworks. I use React every day; this site is built with it. The point is that frameworks are most valuable when you reach for them deliberately, for the things they actually do better than the platform. The platform is catching up fast.
